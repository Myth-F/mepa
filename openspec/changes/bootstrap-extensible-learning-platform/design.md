## Context

The repository contains requirements research and an empty OpenSpec workspace but no application code. The first implementation must therefore establish both product foundations and production operations. The team is small, the MVP deadline is close, new educational modules must be easy to add without code changes, and the complete system must run on a single VPS.

## Goals / Non-Goals

**Goals:**

- Deliver a modular monolith that keeps domain rules independent from Next.js, Prisma, object storage, and future AI vendors.
- Provide a safe, simple block-based authoring and publishing workflow.
- Keep learner and staff identities, sessions, and authorization checks separate.
- Make the full stack reproducibly deployable with Docker Compose on one VPS.
- Preserve PostgreSQL, uploaded media, and backups across container replacement.
- Prepare a trusted, source-aware context boundary for a future contextual tutor.

**Non-Goals:**

- Running an AI model or connecting to an AI provider.
- Multi-node orchestration, Kubernetes, or automatic horizontal scaling.
- A general-purpose rich-text CMS or collaborative editing.
- Public discussions, live chat, or learner-generated module content.
- Shipping a reverse proxy tied to one VPS provider; the application exposes an internal HTTP service for an operator-managed TLS proxy.

## Decisions

### Use a modular Next.js monolith

Use Next.js App Router and TypeScript in one deployable application. Organize business code by capability under `src/modules`, with domain types and use cases depending on ports rather than Prisma, S3, or web framework APIs.

This minimizes deployment and coordination overhead while preserving boundaries that can be extracted later. A split React/NestJS architecture was rejected because it adds an API contract and a second deployment before the MVP benefits from them.

### Store authored modules as immutable published versions

`Module` provides stable identity and slug. `ModuleVersion` stores draft, published, or archived versions. Ordered `ModuleBlock` records use a block type, schema version, position, and JSONB payload. Publishing validates the complete draft in a database transaction, archives the previous published version, and makes the new version immutable.

Each block type is registered with its Zod schema, admin editor, learner renderer, and text projection. This allows a new block type without changing the database schema while preventing arbitrary unvalidated JSON.

Course content is runtime data, not application constants. Public pages read published modules from PostgreSQL and the denormalized search document. Demo seeds are allowed only as an environment initialization path and must use the same authoring domain services and database tables as real authoring; they must not sit in the request path or replace the staff authoring surface.

### Build a custom admin surface with separate staff identity

The same Next.js deployment serves `/admin`, but staff use separate tables, session cookies, and authorization middleware from learners. Staff accounts are provisioned by an administrator; public staff registration is not available. Passwords use Argon2id and random session tokens are stored hashed.

This meets the requested identity separation without operating a second application. Sharing one identity table was rejected because it increases the risk of learner privileges reaching the authoring surface.

All staff routes verify the database-backed staff session in a server layout before rendering. Public navigation is built from verified server-side session state: staff links and labels do not appear in the header, footer, or public landing page for unauthenticated visitors. The staff sign-in route remains reachable only by its direct URL.

### Use a low-friction DSFR-inspired public journey

The public interface follows DSFR information architecture principles: a text-based service identity with a clear baseline, concise direct navigation, visible keyboard focus, skip links, breadcrumbs, plain-language form labels, and explicit empty/error states. It does not reproduce the restricted State brand block or Marianne typeface.

Visitors can start a published module without an account. Account creation is presented as an optional way to retain progress, not as a prerequisite. Technical terms such as “learner account”, “published version”, and “administration” are removed from public-facing copy where they do not help the visitor.

### Abstract media behind S3-compatible storage

The media module exposes a `MediaStoragePort`. Development and the default VPS deployment use MinIO; production can switch to another S3-compatible service through environment configuration. Metadata stored in PostgreSQL includes ownership, object key, MIME type, size, and required alternative text for meaningful images.

Uploads use short-lived signed requests. Object keys are generated by the server and untrusted executable formats are rejected.

### Define but do not activate the AI tutor boundary

Define `TutorProvider` and `ModuleContextBuilder` interfaces. `ModuleContextBuilder` reads only a published version, creates a normalized text projection from registered block types, and retains source identifiers. No provider SDK is installed and the tutor feature flag defaults to disabled.

The future provider adapter will be the only layer allowed to know vendor-specific request and response formats. This prevents an AI integration from becoming a dependency of module authoring or learning progress.

The learner module reader uses a full-width three-column "classroom" layout (progress rail, reading column, assistant column). The right-hand assistant column is the learner-facing surface of this boundary: it is gated by both the feature flag and a configured provider, so the default build renders only an inert, non-interactive placeholder with no input field and no provider request. The interactive chat is mounted there only once the flag is enabled and a provider is configured, after a security review.

### Deploy with a production Docker Compose stack

Provide:

- a multi-stage, non-root application image with a pinned Node.js LTS base and Next.js standalone output;
- `compose.yaml` defining `app`, `postgres`, `minio`, `minio-init`, and `backup`;
- a one-shot `migrate` Compose profile/service that runs `prisma migrate deploy` before an application rollout;
- named volumes for PostgreSQL data, MinIO data, and local backup archives;
- health checks and `depends_on` health conditions;
- internal-only database and MinIO API networks, with only the application HTTP port exposed to the VPS reverse proxy;
- `.env.example` containing names and documentation but no secrets;
- resource-conscious defaults, log rotation, restart policies, and image tags supplied through environment variables.

Migrations do not run automatically in every application replica. The documented deployment sequence pulls/builds the image, runs the one-shot migration, starts services, and verifies health.

Application health separates liveness from readiness. `/api/health` only confirms that the HTTP process can answer and is the Docker healthcheck target. `/api/ready` checks PostgreSQL connectivity for operator monitoring. This avoids Coolify/Docker restart loops when a dependency is temporarily slow while still making dependency failures observable.

The Compose stack intentionally does not terminate TLS. The VPS operator connects Caddy, Traefik, Nginx, or a hosting control panel to the application's loopback-bound port and manages certificates there.

### Make backups and restoration operational requirements

The `backup` service runs scheduled PostgreSQL logical dumps and MinIO mirroring into a persistent backup volume, with configurable retention. The operator runbook documents copying backups off the VPS and performing a restore drill. Local backups alone are not considered sufficient disaster recovery.

## Risks / Trade-offs

- [JSONB block payloads can become incompatible] -> Require a schema version per block, registry validation, migration functions, and fixture tests.
- [Custom administration costs more than a CMS] -> Limit the first editor to known typed blocks, reorder controls, preview, and publication.
- [A single VPS is a single point of failure] -> Keep stateless application images, persistent volumes, health checks, and off-host backups so recovery is reproducible.
- [Compose does not provide zero-downtime database migrations] -> Require backward-compatible expand-and-contract migrations and a documented rollback procedure.
- [MinIO increases VPS operational load] -> Keep the storage port provider-neutral so hosted S3 can replace it by configuration.
- [Separate identities duplicate authentication code] -> Share low-level cryptographic helpers while keeping repositories, cookies, and authorization policies separate.
- [Future AI may expose untrusted content] -> Build context only from published modules and tracked sources; require a future security review before enabling the feature.

## Migration Plan

1. Provision a VPS with Docker Engine, Compose, a TLS reverse proxy, firewall rules, and sufficient disk.
2. Create secrets outside version control from `.env.example`.
3. Start PostgreSQL and MinIO, initialize the bucket, then run the one-shot migration service.
4. Start the application and backup service, configure the reverse proxy, and verify health.
5. Create the initial staff administrator through a one-shot CLI command.
6. Verify a module can be authored, published, consumed, backed up, and restored in staging.

For rollback, retain the previous application image tag and database backup. Roll back application code only when the migration is backward compatible; otherwise restore the matching database and object-storage backup before starting the previous image.

## Open Questions

None for the initial implementation. The VPS reverse proxy product and off-host backup destination remain operator configuration choices and do not change application contracts.
